You are implementing the Chrome extension for the first working iteration of our accessibility-focused voice assistant project.

Context:
- The first iteration already exists.
- It excludes the browser extension, but it already produces structured commands that are intended to be executed by a future extension.
- We are now building that extension.
- Use OpenAI's API only for LLM and voice-related tasks in the wider project, but for this extension itself, keep it deterministic and execution-focused. The extension should NOT call the OpenAI API directly.
- The extension should only receive ready-to-execute commands from the existing app/orchestrator.

Goal:
Build a Chrome extension that can receive structured commands from the existing local app, execute safe browser actions on the active tab, and return structured results back to the app.

High-level product intent:
This project is an accessibility-focused assistive system for users with motor impairments, low vision, and cognitive overload. The extension is the execution layer only. It should feel reliable, predictable, and simple. It should not perform reasoning. It should only interpret structured commands and execute them safely.

What to build:
Implement a Chrome extension (Manifest V3) that supports the following first-iteration capabilities:

1. Ping / health check
- The app should be able to confirm that the extension is installed and responsive.
- Return extension version, active tab URL if available, and basic readiness status.

2. Page context extraction
- Extract visible text from the current page.
- Extract page title.
- Extract current URL.
- Extract interactive elements summary, including buttons, links, inputs, textareas, selects.
- For interactive elements, include lightweight metadata such as:
  - type
  - text/label
  - name
  - id
  - placeholder
  - aria-label
  - disabled state
  - whether element appears visible
Do not return massive raw DOM dumps. Keep it compact and structured.

3. Open / navigate
- Navigate the active tab to a provided URL.
- Also support opening a URL in a new tab if explicitly requested by the command.
- Validate URLs conservatively.

4. Click action
- Accept a structured click command with one or more selector hints.
- Attempt to find the best matching clickable element.
- Click only if confidence is reasonable.
- Return what was matched.
- If no suitable element is found, return candidate matches instead of failing silently.

5. Type into fields
- Accept a structured type/fill command with field hints and values.
- Find the most likely matching input / textarea / contenteditable target.
- Simulate user-friendly input events properly.
- Return which field was matched and filled.
- If ambiguous, return candidates.

6. Read page command support
- Support extraction for “read page” style flows by returning ordered visible text blocks.
- Do not do speech synthesis inside the extension yet.
- The app/backend can decide how to use the text.

7. Highlight target elements
- When an element is clicked, typed into, or selected as a candidate, visibly highlight it briefly.
- Use an accessible high-contrast temporary outline.

8. Structured command execution
- The extension should accept commands that are already planned by the orchestrator.
- It should execute one command at a time.
- It should return structured results in a stable schema.

Important architectural rule:
The extension must be dumb and reliable.
- No LLM calls.
- No OpenAI API calls.
- No hidden planning.
- No guessing beyond lightweight element ranking logic.
- No silent dangerous actions.

Scope constraints for this first extension iteration:
- Chrome only.
- Local development only.
- Do not support arbitrary desktop control.
- Do not support file system access.
- Do not support sending messages or submitting forms automatically unless the command explicitly says so and confidence is high.
- For now, do not auto-submit forms by default.
- Prefer filling and highlighting over final submission.

  1. Extension popup or sidepanel that can manually connect to localhost API
  2. Background service worker polling or requesting localhost
  3. Chrome native messaging is NOT required
Choose the simplest reliable localhost integration for a hackathon MVP.

Command schema expectations:
Design the extension around stable structured commands.
Support commands like these, or a very similar schema:

{
  "id": "cmd_123",
  "type": "get_page_context"
}

{
  "id": "cmd_124",
  "type": "navigate",
  "url": "https://example.com",
  "newTab": false
}

{
  "id": "cmd_125",
  "type": "click",
  "target": {
    "text": "Contact Us",
    "role": "button",
    "selector": null
  }
}

{
  "id": "cmd_126",
  "type": "fill_field",
  "target": {
    "fieldHint": "email"
  },
  "value": "jaime@example.com"
}

{
  "id": "cmd_127",
  "type": "fill_form",
  "fields": [
    { "fieldHint": "name", "value": "Jaime Da Cruz" },
    { "fieldHint": "email", "value": "jaime@example.com" }
  ]
}

Result schema expectations:
Return structured results like:

{
  "commandId": "cmd_126",
  "ok": true,
  "action": "fill_field",
  "matched": {
    "tag": "input",
    "name": "email",
    "id": "email",
    "label": "Email Address"
  },
  "message": "Filled email field successfully"
}

And for ambiguous cases:

{
  "commandId": "cmd_125",
  "ok": false,
  "action": "click",
  "message": "No high-confidence match found",
  "candidates": [
    { "text": "Contact", "role": "link" },
    { "text": "Contact Us", "role": "button" }
  ]
}

Element matching logic:
Implement lightweight ranking heuristics. Prefer deterministic ranking over complex magic.
For inputs:
- exact name match
- exact id match
- exact aria-label match
- exact associated label text match
- placeholder contains field hint
- partial matches after normalization

For clickable elements:
- exact visible text match
- aria-label match
- title match
- role match if available
- selector match if provided
- partial normalized text match

Normalize comparisons:
- lowercase
- trim whitespace
- collapse spaces
- ignore punctuation where reasonable
