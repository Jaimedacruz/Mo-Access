You are working inside the existing accessibility-focused hackathon project.

Your task is to update and improve the **feedback layer** so that spoken feedback uses the **OpenAI text-to-speech API** instead of the browser's native `speechSynthesis` voices.

Important context:
- The orchestrator already exists.
- The Chrome extension already exists.
- A first feedback layer may already exist.
- Do NOT redesign the architecture.
- Do NOT rebuild unrelated parts.
- Focus on upgrading the spoken feedback path while keeping the system accessible, clean, and demo-ready.
- Use OpenAI's API for all voice-related tasks in this project.

## Goal
Replace or abstract the current browser-native TTS path with an **OpenAI-powered speech output layer** that produces better spoken feedback quality while keeping the UX calm, concise, and accessibility-first.

The user should still get:
1. **Visual feedback** during task execution
2. **Optional voice feedback**
3. Clear status transitions so the user always understands what the system is doing

The spoken feedback should feel higher quality than default browser TTS, but the product should remain practical for a hackathon demo.

---

## High-level requirements

### 1. Use OpenAI text-to-speech for spoken feedback
Implement spoken feedback through the OpenAI API rather than `speechSynthesis`.

Requirements:
- Route spoken status updates through the backend/orchestrator or the most appropriate secure layer
- Do **not** expose API keys in frontend or extension code
- Generate audio from text using OpenAI TTS
- Return audio safely to the frontend for playback
- Keep the implementation simple and reliable

### 2. Keep voice feedback optional
There should still be a UI toggle to enable or disable spoken feedback.

Requirements:
- Voice feedback should remain OFF by default unless the current product already has a clear accessibility preference system that suggests otherwise
- The UI toggle should remain easy to understand
- Spoken feedback should only occur for meaningful status updates, not every micro-event

### 3. Preserve the visual feedback system
Do not remove the current visual status layer.

The product should still clearly show messages such as:
- "Opening page..."
- "Looking for the form..."
- "Filling the email field..."
- "Ready to submit. Waiting for confirmation."
- "Done."
- "I couldn't find the button."

### 4. Integrate OpenAI TTS with feedback events
The existing feedback event system should be used, or extended if needed.

Support event types such as:
- info
- progress
- success
- warning
- error
- awaiting_confirmation

Only some events should trigger spoken output.

## Spoken feedback behavior

### Speak only meaningful updates
Examples of messages that SHOULD usually be spoken:
- "Opening the page now."
- "Filling the form now."
- "I found the submit button."
- "I am ready to submit the form. Please confirm."
- "I couldn't find that field."

Examples that SHOULD NOT usually be spoken:
- every tiny internal execution detail
- repeated duplicate messages
- noisy debug information

### Avoid excessive chatter
Implement guardrails such as:
- de-duplication of repeated messages
- minimum spacing between spoken updates if needed
- cancellation/replacement strategy if a new higher-priority message arrives
- prevent overlapping audio playback

### Confirmation prompts must be spoken clearly
For risky actions like:
- submitting a form
- sending a message
- navigating away from unsaved content

The system should speak a concise confirmation prompt such as:
- "I am ready to submit the form. Please confirm."
- "I wrote the message and I am ready to send it. Please confirm."

## Security and architecture requirements

### 1. No API key exposure
The OpenAI API key must never appear in:
- frontend source
- extension source
- browser-exposed configs

Use the backend/orchestrator or another secure server-side layer to call OpenAI.

### 2. Add a small server-side TTS endpoint or service
Implement a secure backend path that:
- accepts short text for speech
- validates input
- calls OpenAI TTS
- returns playable audio to the frontend

Keep it simple and production-minded.

### 3. Keep the interface modular
Create or refine a small speech service abstraction so that the UI does not care whether speech came from:
- OpenAI TTS
- or a fallback provider if you decide to keep one behind the abstraction

Suggested shape:
- `speakFeedback(text, options?)`
- backend `synthesizeFeedbackAudio(text, options?)`

Do not force these exact names if the repo already has a better pattern.
