You are helping build the first iteration of an accessibility-focused hackathon project.

Project context
- This project is inspired by OpenClaw’s orchestration pattern, but it must NOT copy or depend on the full OpenClaw repo.
- We are building our own lightweight, self-contained orchestration layer.
- The product goal is an AI-powered assistive system that helps disabled users perform browser-based tasks more easily through voice.
- For this first iteration, EXCLUDE the browser extension entirely.
- Instead, produce structured commands that are ready to be consumed later by a browser extension.
- We will use OpenAI’s API for all LLM and voice-related tasks.
- This first iteration must be realistic for a hackathon, simple, clean, and runnable locally.

What to build
Build a local MVP that supports this flow:
1. User speaks or records a voice command in the web app
2. The app sends audio to OpenAI speech-to-text
3. The transcript is shown to the user
4. The transcript is sent to an orchestrator API
5. The orchestrator uses an OpenAI model to convert the transcript into a structured intent
6. The orchestrator turns the intent into a step-by-step action plan
7. The app displays:
   - transcript
   - parsed intent
   - planned steps
   - status messages
8. The final output should be “browser-extension-ready commands”, not real browser automation

Important scope constraints
- DO NOT build a browser extension yet
- DO NOT implement real DOM interaction
- DO NOT implement OpenClaw itself
- DO NOT add unnecessary infrastructure
- DO NOT add authentication
- DO NOT add databases 
- Keep the system local-first and self-contained
- The code should be clear, minimal, and demo-friendly

Primary user value of iteration 1
This first version should already demonstrate the key idea:
- voice input
- AI understanding of user intent
- AI decomposition of a browser task into safe structured actions
- accessibility-focused interaction flow

Recommended stack
Use a simple monorepo or clean project structure with:
- a frontend web app

Preferred technologies

- Backend: Node.js + TypeScript, lightweight API routes or Express
- Validation: Zod
- OpenAI API for:

2. Backend API
Create API endpoints for:

- command parsing / planning

3. Intent parsing
Convert natural language into a structured intent schema.
Use a strict JSON schema and validate it.

Suggested intent types:
- open_page
- fill_form
- read_page
- compose_message
- search_web

Suggested intent schema shape
Use TypeScript + Zod and validate returned data.
Example structure:
{
  "type": "fill_form",
  "page": "support",
  "fields": {
    "name": "Jaime Da Cruz",
    "email": "jaime@example.com"
  },
  "requiresConfirmation": true,
  "safetyLevel": "medium"
}

4. Action planning
Translate the intent into browser-extension-ready step commands.
Use a second explicit planning step in code.
The planner should output structured steps like:
{
  "steps": [
    { "type": "navigate", "target": "/support" },
    { "type": "type", "fieldHint": "name", "value": "Jaime Da Cruz" },
    { "type": "type", "fieldHint": "email", "value": "jaime@example.com" },
    { "type": "confirm", "message": "Ready to submit the support form." }
  ]
}

Suggested step command types
- navigate
- click
- type
- extract_text
- confirm
- search
- compose_message

5. Safety model
Mark risky actions clearly.
Examples of actions that should require confirmation:
- sending messages
- submitting forms
- navigating away from a form after filling
- any action that could have side effects

The backend should attach safety metadata such as:
- requiresConfirmation
- safetyLevel
- confirmationMessage

6. Status and transparency
The user should always see what the assistant is doing.
Represent states like:
- idle
- transcribing
- parsing
- planning
- ready
- error

Implementation guidance
Build the project in a way that is easy to extend later with a browser extension.
That means:
- keep the planner independent from execution
- define clean command schemas now
- do not mix UI logic with orchestration logic

Prompting requirements for the LLM
Use a strong system prompt for intent parsing.
The model must:
- classify the intent into one of the supported intent types
- extract relevant fields
- infer confirmation requirements conservatively
- return JSON only
- avoid hallucinating unsupported actions
- prefer safe outputs over ambitious ones

Suggested parser behavior
